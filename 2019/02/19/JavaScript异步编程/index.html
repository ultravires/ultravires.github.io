JavaScript异步编程

JavaScript 语言的执行环境是“单线程（Single thread）”的。线程（Thread）是操作系统能够进行运算调度的最小单位。一条线程指的是进程中一个单一顺序的控制流。所以，“单线程”表示一次只能完成一件事。如果有多个任务就必须排队。

这种模式（同步）的好处是实现起来比较简单，执行环境相对单纯；坏处是显然存在死等的可能性，如果前一个任务耗时较长，则后面的任务就必须等待，会拖延整个程序的执行。常见的浏览器无响应往往就是因为某一段 JavaScript 代码长时间运行（比如死循环）而导致整个页面卡在某个地方，其它任务无法执行。

为了解决这个问题，JavaScript 语言将任务的执行模式分成了两种：同步（Syschronous）和异步（Asynchronous）

“同步模式”就是指前面的顺序执行，后一个任务等待前一个任务执行结束；“异步模式”则不同，每一个任务有一个或多个回调函数，回调函数后的任务可以不等回调函数执行完毕就可以先执行，程序的执行顺序与代码的先后排列顺序是不一致的。

“异步模式”非常有用。在浏览器端，耗时很长的操作都应该异步执行，避免浏览器失去响应，最好的例子就是 Ajax 操作。

下面是“异步模式”编程的几种方法。

回调函数

“回调函数”就是指被作为实参传入另一函数，并在该外部函数内被调用以用来完成某些任务的函数。

“回调函数”可以是同步的也可以是异步的。而并不是新手所认为的只要是回调函数就一定是“异步模式”。

采用“回调函数”实现“异步模式”的优点是简单、容易理解和部署，缺点是不利于代码的阅读和维护，各个任务之间高度耦合（Coupling），流程较为混乱，而且每个任务只能指定一个回调函数。

事件监听

事件驱动模式下任务的执行不取决于代码的顺序，而取决于某个事件是否发生。

发布/订阅

发布/订阅模式（publish-subscribe pattern）又称观察者模式（observer pattern）。

引用一下阮一峰老师的例子：我们假定存在一个“信号中心”，某个人物执行完成，就向信号中心发布一个信号，其他任务可以向信号中心订阅这个信号，从而知道自己什么时候可以开始执行。

再举一个比较生动的例子：微信公众号，微信公众号可以“发布”文章，用户可以“订阅”公众号，当然用户也有“取消订阅”公众号的权利。所以“订阅者”也可以取消订阅某个事件，这样就算你发布了这个事件我也不会理会。当然，用户还可以选择获取什么样的文章，所以“发布”是可以传参的。（你可能光看例子会很蒙，建议仔细分析代码后再结合例子推敲。）

事件中心：

我们每订阅一条事件就向事件中心添加一条，每取消订阅一次就移除一条，每发布一次信息就执行一条。
